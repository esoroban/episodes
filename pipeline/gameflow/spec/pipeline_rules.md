# Pipeline Rules — Связь gameflow с другими слоями

## Карта инвокации: правило → инструмент

Каждое правило имеет один или несколько способов запуска. Если правило
есть, а инвокации нет — это пробел, его нужно закрывать (validator или skill).

### Автоматическая валидация (`python3 tools/validate_gameflow.py`)

| # | Что проверяет | Файл-чек |
|---|---|---|
| 1 | Дубликаты ключей в YAML | строгий парсер |
| 2 | Битые переходы (next_default/next_fail/merge_to) | граф сцен |
| 3 | Недостижимые сцены | обход графа |
| 4 | Неиспользуемые флаги | set vs require |
| 5 | Персонажи без интродукции | characters_present |
| 6 | Полнота квизов (≥1 correct) | options |
| 7 | Префикс «Софа:» в feedback | feedback_success/soft_fail |
| 8 | voice_message без who/line/duration | unlock_button.reveals |
| 9 | Софа описывает звук («слышишь», «треск») | dialogue Софы |
| 10 | Утечка «Софа: ...» в author_text | author_text/_after |
| 11 | Имя персонажа до первого появления | cross-episode |
| 12 | Отсутствует source_ref | каждая сцена |
| 13 | Запрещённый блок `previously:` | top-level |
| 14 | Невалидный branch_type | scenes[].branch_type |
| 15 | Софа говорит, но не в `characters_present` | `dialogue[]` / `dialogue_after[]` with `who: Софа` |
| 16 | Self-loop в `next_fail` | `next_fail == scene_id` |
| 17 | Именованное место до его введения (`Зеркальный Город` и т.п.) | текст эпизодов кумулятивно |
| 18 | Live-персонаж **говорит** в chat-сцене | `dialogue[].who` / `dialogue_after[].who` |
| 19 | Ритм веток — ≥1 choice-точка на эпизод | `branch_type` / `scene_type: choice` |
| 20 | В HTML chat-сцены нет внешнего `author-text` рядом с `.phone` | HTML scene sections |

**Запуск перед каждым деплоем.** Падает на error — деплой блокируется.

### Скиллы (вызываются по триггерам)

| Триггер / задача | Скилл | Когда вызывать |
|---|---|---|
| Создать план эпизода из дня | `episode-plan` | новый день, ещё нет плана |
| Привязать квизы к сюжету | `episode-map` | после `episode-plan` |
| Собрать gameflow YAML | `gameflow-build` | после `episode-map` |
| Восстановить драму из книги | `gameflow-drama` | после `gameflow-build`, если драма ушла |
| Добавить ветвления | `gameflow-branch` | после `gameflow-drama`, если эпизод линеен |
| Проверить ссылки на персонажей/факты/термины | `qa-references` | после любых правок текста |
| Проверить замкнутость веток | `qa-branches` | после `gameflow-branch` |
| Проверить эпизод на ответ-до-вопроса, теорию, чистоту | `qa-episodes` | финальный QA контента |
| Проверить чистоту lesson briefs | `qa-briefs` | после `lesson-brief` |
| Опубликовать на Render | `deploy-game` | финал, после всех QA |

**Триггер-фразы скилла указаны в его SKILL.md** (поле `description`).

### Рендер (`python3 tools/build_game.py`)

Рендерер enforce'ит правила доставки:
- Софа в `characters_present` + говорит → автоматически Telegram-режим
- `who: автор` → voice-over карточка (никогда чат-пузырь)
- `unlock_button.reveals.type: voice_message` → отдельная voice-плашка
- `interactions[]`/`followup_interaction` → inline-keyboard квизы в чате

### Что НЕ автоматизировано (контролируется через qa-* скиллы)

- 100% переноса контента из book/source — `qa-episodes`
- Глубина веток ≤ 3 сцены — `qa-branches`
- Flavor_detour не содержит plot-critical info — `qa-branches`
- Ответ к квизу не появляется до квиза — `qa-episodes`
- Quiz count == план в day_NN.yaml — `qa-episodes`

Если правило здесь — но скилл его не проверяет, открывай PR на новый Check
в `validate_gameflow.py` или новый раздел в соответствующем скилле.

## Архитектура слоёв

```
pipeline/source/episodes/          ← source of truth (YAML планы эпизодов)
        ↓
pipeline/gameflow/episodes/ ← игровой scene-flow (этот слой)
        ↓

publish/game/               ← HTML-превью
pipeline/gameflow/visuals/  ← visual briefs для художки
```

## Source layer: поле `branch_intent` (опциональное)

Чтобы автор мог **объявить развилки на уровне плана** (а не открывать их
только на gameflow-этапе), в `pipeline/source/episodes/day_NN.yaml` в
каждом эпизоде допускается поле `branch_intent` — список выборов, которые
должны появиться в gameflow.

Формат:
```yaml
- ep: 2
  branch_intent:
    - where: "s08 — чат про «выдумал сестру»"
      type: "flavor_detour"
      options: ["walk_away", "argue"]
      purpose: "реакция Марко на публичное унижение"
      merge_to: "s09"
    - where: "s32 — Вера просит дневник"
      type: "gated_response"
      options: ["keep", "give"]
      purpose: "проверка доверия к Вере; дневник больше не load-bearing после ep_002"
      merge_to: "s33"
```

Поля элемента `branch_intent`:

| Поле | Тип | Обязательно | Описание |
|------|-----|-------------|----------|
| `where` | string | да | На какую сцену gameflow навешивается выбор. Формат: `sNN — краткий маркер` |
| `type` | enum | да | Один из допустимых `branch_type` (`flavor_detour`, `gated_response`, `cosmetic_branch`, `soft_fail_loop`) |
| `options` | list[string] | да | ≥2 идентификаторов веток. Короткие id в snake_case |
| `purpose` | string | да | Зачем эта развилка нужна — одна фраза. Нужно, чтобы `gameflow-build` понимал тон |
| `merge_to` | string | да | В какую сцену сходятся ветки. Обычно следующая после `where` |

**Как это читает `gameflow-build`:**
1. Берёт каждый элемент `branch_intent` и создаёт:
   - сцену `ep00X_sNN_choice` (drama, scene_type: choice) с options[], указывающими на детуры
   - по одной сцене `ep00X_sNN_<option_id>` на каждую опцию (branch_type = type)
   - `merge_to` на указанную сцену
2. Выставляет `next_default` оригинальной сцены на `sNN_choice`.
3. Содержимое веток (author_text) автор дописывает вручную или через `gameflow-branch`.

**Ритм веток.** Правило: **минимум 1 `branch_intent` на эпизод**. Валидатор
Check 19 проверяет, что в каждом ep-блоке `day_NN.yaml` есть хотя бы один
`branch_intent` ИЛИ в соответствующем `ep_NNN.yaml` есть хотя бы одна
сцена с `branch_type` / `scene_type: choice`.

**Почему не отдельный слой.** `branch_intent` — это аннотация в существующем
source-файле, а не новый формат. Скиллы без `branch_intent` работают как
раньше. Никакой тройной синхронизации.

## Отношение к pipeline/source/episodes

| Правило | Описание |
|---------|----------|
| Source of truth | `pipeline/source/episodes/day_*.yaml` — единственный источник сюжета и уроков |
| Ссылки обязательны | Каждая сцена в gameflow содержит `source_ref`, указывающий на блок в episodes |
| Не дублировать контент | Gameflow не повторяет полный текст из episodes — он структурирует его в сцены |
| Не противоречить | Gameflow не может менять сюжет, порядок терминов или quiz-логику из episodes |

## Что gameflow ДОБАВЛЯЕТ поверх episodes

1. **Разбиение на сцены** — episodes описывают блоки (drama, sofa_block, challenge), gameflow дробит их на отдельные экранные моменты.
2. **Навигация** — `next_default`, `next_fail`, `merge_to` — явный граф переходов.
3. **Ветвления** — soft fail loops, flavor detours (в episodes их нет).
4. **Visual briefs** — конкретные кадры для каждой сцены.
5. **UI-режимы** — `reading`, `quiz`, `choice`, `dialogue`.

## Что gameflow НЕ делает

- Не переписывает авторский текст — берёт из episodes/book.
- Не добавляет новые quiz — только структурирует существующие.
- Не создаёт новых сюжетных поворотов — только декоративные ветвления.
- Не определяет визуальный стиль — только описывает, что рисовать.

## Отношение к server/book

`server/book/ep_*.md` — развёрнутый авторский текст. Gameflow может ссылаться на него для полного текста сцен, но не зависит от него.

Направление: `episodes → gameflow → book` (gameflow не читает book).

## Отношение к publish/game

`publish/game/ep_*.html` — один из возможных рендеров gameflow. HTML генерируется из gameflow, не наоборот.

Направление: `gameflow → publish/game` (однонаправленная генерация).

## ПРАВИЛО: 100% переноса контента (обязательно)

**Всё содержимое `server/book/ep_NNN.md` должно быть перенесено в gameflow YAML. Без исключений.**

| Запрещено | Допустимо |
|-----------|-----------|
| Пропускать quiz-вопросы из источника | Разбивать один quiz на две сцены |
| Сокращать или убирать нарративные сцены | Добавлять branch/detour сцены поверх |
| Изобретать события, которых нет в источнике | Менять порядок в пределах логики рендера |
| Объединять несколько quiz в один вопрос | — |

**Правило `previously`:** Блоки `previously` ЗАПРЕЩЕНЫ. Ребёнок играет подряд,
ему не нужен рекап. Никаких "ранее", "в прошлый раз", "помнишь" для пересказа
предыдущих событий.

**Правило сохранения драмы (обязательно):**

100% перенос контента из `server/book/ep_NNN.md`. Книга + план уже содержат
всю драму и все квизы в правильной пропорции. Gameflow — это оформление,
не переписывание. Если в книге между квизами есть нарратив — он должен быть
в gameflow. Подробности — skill `gameflow-drama`.

**Chat-UI занимает экран целиком (обязательно):**

В chat-сцене (phone-rail, Telegram-интерфейс) на экране **только phone**.
Никаких параллельных панелей рядом или под ним — ни `author_text` блоков,
ни иллюстраций, ни CTA-кнопок вне чата.

Весь авторский текст (`author_text` И `author_text_after`) в chat-сцене
рендерится **внутри** phone-rail как `.voice-row` карточки (голос автора
между сообщениями Софы). Это делается в `build_chat_messages` автоматически.

**Запрещено:** рендерить в chat-сцене что-либо через
`<div class="author-text">…</div>` вне `.phone`. Это двойной рендер
(текст окажется И внутри чата как voice-over, И снаружи отдельным
блоком) и нарушает правило «phone = весь экран».

Инцидент: `ep_001 s04a` (`branch_type: gated_response`, про мороженое)
рендерился дважды — внутри чата и ниже отдельной панелью — после того
как branch-сцены стали рендериться standalone (см. фикс абсорбции веток,
2026-04-19). `render_scene` исправлен на единичный вывод через
`render_chat`.

**Chat-UI vs Drama — кто допустим в каком режиме (обязательно):**

В Telegram-chat-UI (phone-rail, chat-messages, chat-input-bar, voice-message-бабл внутри чата) допустимы **только три «спикера»**:

1. **Софа** — text-bubbles (её канал)
2. **Марко** — outgoing text или voice-message, если печатает/говорит Софе в чат
3. **автор** — voice-over card между сообщениями (голос за кадром, не bubble)

**Все остальные персонажи (мама, Вера, Данила, Олена, Витя, Лина, София-сестра, Макс, Леон, Сэм, Рэй и др.) — ВНЕ чата.** Никогда не как text-bubble, не как отправитель, не как voice-message в `.chat-messages`. Их реплики всегда в drama-dialogue-блоке отдельной сцены.

**Исключение:** `unlock_button.reveals.type: voice_message` в `ep_001 s11` — разблокированная запись Софии. Это не диалог, а найденный артефакт (как прикреплённый файл в чате).

**Следствие — правило разделения сцен:**

Если в одной сюжетной точке нужны и drama (живые персонажи), и комментарий Софы, сцена разбивается на две последовательные:

| Сцена | `characters_present` | Что в ней |
|---|---|---|
| `ep00X_sNN` (drama) | `[Марко, X_live]` | author_text, dialogue живых персонажей. Софы нет. |
| `ep00X_sNN_sofa` (chat) | `[Марко, Софа]` | Dialogue Софы, квизы, unlock. Живых персонажей нет. |

`next_default` первой → вторая. `next_default` второй → что было у первой.

**Таблица правил рендера:**

| Правило | Описание |
|---------|----------|
| **Софа = ВСЕГДА Telegram** | Если Софа говорит — сцена рендерится как чат. В `characters_present` только `[Марко, Софа]` (и опционально `автор` как voice-over). |
| **Live-персонажи = НИКОГДА Telegram** | Если в `characters_present` есть кто-либо не из `{Софа, Марко, автор}` — сцена рендерится как drama, Софы в ней быть не может. |
| **Split-сцены** | Для связки drama → комментарий Софы — создавать две сцены. См. таблицу выше. |
| Запрет на «Софа: ...» в `author_text*` | Реплики Софы только в `dialogue`/`dialogue_after`, не в `author_text`. |
| Вера = всегда drama | Signature-локация Веры — школьный коридор. Её сцены никогда не чат. |
| `who: автор` в чате | Допустимо как voice-over card (`.vo-card`) между репликами Софы. |
| `correct_logic` | Не рендерится в чате; хранится для скрытого `quiz-explanation`. |
| Нет дублей | `feedback_success` и `correct_logic` не дублируют друг друга. |
| Разрыв chain | Чтобы разорвать phone chain — drama-сцена между chat-сценами или смена `location`. |
| Длина chain | Не ограничена. Пока идёт Софа и нет живых персонажей — chain любой длины. |

Проверка: chain рвётся там, где драма этого требует сюжетно. Не искусственно
по количеству сообщений.

**Каналы доставки контента (voice vs text) — обязательно:**

Игра доходит до игрока двумя каналами: **голосом** (voice-over) и **текстом**
на экране телефона (чат-пузыри). Интерфейс — один экран телефона; всё остальное
либо озвучивается, либо показывается иллюстрацией поверх экрана.

| Говорящий | Канал | Условие |
|-----------|-------|---------|
| `Софа` | text | всегда — Софа НЕ озвучивается, только чат |
| `автор` | voice | всегда — автор НЕ виден в интерфейсе |
| `Марко` | text | если Софа ∈ `characters_present` (Марко печатает ей в чат) |
| `Марко` | voice | иначе (Марко говорит вслух в сцене) |
| Любой другой персонаж | voice | всегда (мама, Вера, Лина, Макс, Леон, Рей, Сем, Данила и т.д.) |
| `voice_message` (unlock_button reveals) | voice | исключение — разблокированная запись (напр. София в ep_001 s11) |

**Запреты в тексте YAML:**

- Не описывать звучание Софы: «голос Софы», «треск», «электронный», «ровный голос», «шипение», «обрыв голоса» — у Софы нет звука.
- В репликах Софы избегать слов, предполагающих слух: «слышишь», «услышь», «звучит» — заменять на «читаешь», «видишь».
- `author_text` не описывает звук телефона (треск, щёлчки, электронный звук). Допустимо: «экран мигает», «экран гаснет», «телефон вибрирует» (вибрация — тактильно, не звук).

**«Голос» с большой буквы** — это антагонист, отдельный персонаж (громкоговорители
в Зеркальном Городе). Озвучивается. Не путать с «голосом Софы».

**Правило рендера квизов — Telegram-бот inline keyboard (обязательно):**

Все квизы (`interaction_type: vote`, `interaction_type: choice` с Софой) рендерятся
в чате как сообщения Telegram-бота с прикреплённой inline-клавиатурой.
Не отдельным экраном.

| Элемент квиза | Рендер |
|---------------|--------|
| `question` | Пузырь Софы (как обычное сообщение `in`), но без нижнего отступа — клавиатура примыкает снизу |
| `options[]` | Inline-keyboard под пузырём: кнопки в столбик (или рядом, если короткие), стиль бот-кнопок (светлая заливка, синий текст, скругление ~10px) |
| tap по правильной опции | Кнопка подсвечивается зелёным с галочкой, клавиатура блокируется, Софа шлёт `feedback_success` как следующее сообщение |
| tap по неправильной опции | Кнопка подсвечивается красным коротко, Софа шлёт `feedback_soft_fail`, **тот же вопрос заново инжектится новым пузырём ниже**, tapped-кнопка в старом пузыре дизейблится. Игрок отвечает, пока не правильно |
| Переход дальше | После правильного ответа — автоматически следующий шаг скрипта (следующая реплика Софы или CTA в drama) |

**Важно:** retry реализует рендер, а не YAML-граф. `next_fail` **не должен** указывать на `scene_id` той же сцены (см. Check 16). Если нужен «мягкий тупик» другой сценой — используй `branch_type: soft_fail_loop` с отдельным `scene_id`.

**Запреты:**
- Квиз НЕ рендерится как отдельный fullscreen экран с вариантами ответов
- Варианты ответов НЕ показываются как обычные пузыри Марко — только как inline-keyboard
- `correct_logic` в чате не показывается (это для внутреннего разбора)

**Исключение — `life_question` и особые сцены:** когда по замыслу квиз нужно вывести
из контекста чата (например, рефлексия Марко в драма-сцене) — допустимо отдельным
экраном, но это должно быть явно отмечено в gameflow YAML полем `render_as: drama_quiz`.

**CTA-кнопка из чата в драму:** когда сцена требует ухода из Telegram в drama-сцену
(иллюстрация + voice-over), в конце чат-сегмента рендерится CTA-кнопка —
центрированная пилюля с hint-подписью сверху («СОФА ЗАДАЁТ ВОПРОС», «ДАЛЬШЕ — СЦЕНА»)
и акцентной кнопкой («Перейти к вопросу →», «Продолжить →»). Клик переключает
UI-режим на drama.

**Слот автора в чате (debug vs production):**

Автор — voice-over. Для вычитки контента до записи озвучки нужен визуальный слот,
который видно в тексте. В рендере:

- **Debug mode (вычитка):** между репликами Софы рендерится карточка `.vo-card` с
  меткой «АВТОР», пульсирующей точкой и полным текстом `author_text` / `author_text_after`
  из YAML. Стиль отличается от пузырей (курсив, левая полоса, центрирование, полупрозрачный фон).
- **Production mode (релиз):** та же карточка схлопывается в тонкую рельсу с меткой
  `АВТОР (voice-over)` — текст скрыт (играет только озвучка). Слот зарезервирован,
  чтобы при переключении режима ничего не двигалось.

Переключение — флагом `debug_author_text` (по умолчанию ON до релиза, OFF в проде).

**Артефакты для downstream: per-episode JSON + manifest.json (обязательно):**

На каждый эпизод `build_game.py` эмитит ДВА артефакта из одной data model:

| Файл | Для кого |
|---|---|
| `server/game/ep_NNN.html` (или `uk/ep_NNN.html`) | Игрок читает в браузере |
| `server/game/ep_NNN.json` (или `uk/ep_NNN.json`) | Downstream: озвучка (TTS), картинки (image prompts), любой парсер |

Плюс один глобальный индекс:

| Файл | Содержит |
|---|---|
| `server/game/manifest.json` | Список всех эпизодов, обе языковые версии, content-hash каждого JSON |

**Структура per-episode JSON** (ключевое):

```json
{
  "schema_version": 1, "episode_id": 1, "lang": "ru",
  "title": "...", "lesson": "1A.1",
  "terms_introduced": [...], "terms_used": [...],
  "enter_requires": {...},
  "scene_count": N, "quiz_count": N, "branch_count": N,
  "scenes": [
    {
      "id": "ep001_s01", "kind": "narrative", "is_chat": false,
      "location": "...", "time": "...", "chars": [...],
      "mood": "...", "branch_type": null,
      "set_flags": [...], "require_flags": [...],
      "nav": {"next": "...", "next_success": null, "next_fail": null, "merge_to": null},
      "text": [
        {"who": "author", "line": "..."},
        {"who": "Марко", "line": "..."},
        {"who": "мама", "line": "..."}
      ],
      "quizzes": [{"question": "...", "options": [...], "feedback_success": "...", "feedback_soft_fail": "...", "correct_logic": "..."}],
      "choice": null,
      "unlock": null,
      "visual_brief": {...},
      "source_ref": "day_01.ep1.drama"
    }
  ]
}
```

- **`text[]`** — упорядоченная последовательность реплик (author / speaker / Sofa / Marko). Воспроизводит порядок из YAML: `author_text` → `dialogue[]` → `author_text_after` → `dialogue_after[]`. Каждый элемент — один атомарный speech-act.
- **`who`**: `"author"` (narrator), `"sofa"` (bot), `"marko"` (protagonist), либо имя персонажа (`"Лина"`, `"Вера"`, `"мама"` и т.д.).
- **`is_chat`**: `true` если сцена рендерится как Telegram-UI (Sofa chat). Озвучка может по этому флагу менять стиль TTS для Sofa.
- **`quizzes[]`**: массив, потому что сцена может содержать batch (`interactions[]`) из нескольких квизов.
- **`visual_brief`** — берётся из RU в обеих языках (тех-поле для художника).

**Контракт «HTML ≡ JSON»** (инвариант, проверяется автоматически):

Текст, видимый игроку в HTML, совпадает с текстом в `text[]`, `quizzes[].question/options/feedback`, `choice.question/options`, `unlock.button_text/reveals.line` в JSON.

Не сравниваются: `correct_logic` (скрытое поле), `visual_brief` (dev-метаданные),
заголовки эпизода и end-screen (UI chrome), debug-бургер.

Инвариант гонит `tools/validate_artifacts.py`:
```
python3 tools/validate_artifacts.py           # все эпизоды, RU + UK
python3 tools/validate_artifacts.py ep_001    # один эпизод (обе языка)
python3 tools/validate_artifacts.py --lang ru # только RU
```

На drift — exit code 1 с точным diff'ом, где разошлось.

**Зачем:** автор ревьюит в HTML. Озвучка/картинки потребляют JSON. Если артефакты разойдутся — approved правки пропадут в downstream. Инвариант делает drift невозможным: оба выходят из одной in-memory структуры build_game.py → проверка срабатывает, если кто-то разжёг руками один артефакт в обход билда.

**Правило интеграции downstream:**

Пайплайны озвучки и картинок **никогда не парсят HTML**. Они читают:
1. `server/game/manifest.json` — discovery (какие эпизоды есть, хеши).
2. `server/game/ep_NNN.json` (или `uk/ep_NNN.json`) — содержание.
3. По `content_hash` в manifest определяют, что обновилось.

**UK-слой (локализация) — оверлей, не отдельный формат (обязательно):**

Украинский перевод лежит в `pipeline/gameflow/episodes_uk/ep_NNN.yaml`
как **текстовый оверлей**: повторяет структуру RU-эпизода, но содержит
ТОЛЬКО переводимые поля. Вся структура (scene_id, навигация, branches,
visual_brief, characters_present, флаги) берётся из RU при сборке.

**Что переводится:**
- `episode_title`, `terms_introduced`
- `author_text`, `author_text_after`
- `dialogue[].line`, `dialogue_after[].line` (порядок совпадает с RU,
  merge по индексу; `who` остаётся для валидации)
- `question`, `correct_logic`, `feedback_success`, `feedback_soft_fail`
- `options[].text` (merge по `id`)
- `interactions[].*` (индекс) и `followup_interaction.*`
- `unlock_button.text` и `unlock_button.reveals.line`
- `mood` (опционально)

**Что НЕ переводится / не лежит в оверлее:**
- `scene_id`, `next_default`, `next_success`, `next_fail`, `merge_to`
- `branch_type`, `set_flags`, `require_flags`, `enter_requires`
- `visual_brief` (источник для картинок; переводится отдельно, если
  визуальный пайплайн UK-специфичен — но по умолчанию art-брифы
  остаются RU как технический язык)
- `scene_type`, `interaction_type`, `source_ref`
- `location`, `time`, `characters_present`

**Сборка:**
```
python3 tools/build_game.py --lang uk          # все UK-эпизоды → server/game/uk/
python3 tools/build_game.py --lang uk ep_001   # один эпизод
```

`load_episode_lang()` подмешивает overlay в RU data model. Рендер
использует merged data. Если overlay отсутствует — эпизод пропускается
(с предупреждением).

**Debug-бургер: навигация по сценам и эпизодам (обязательно):**

Каждый эпизодный HTML обязан содержать бургер-меню (иконка `☰` в правом
верхнем углу) с двумя секциями:

1. **Эпизод** — список всех существующих эпизодов (1…N) с ссылками
   на соответствующие `ep_NNN.html`. Текущий эпизод выделяется.
2. **Сцены этого эпизода** — список всех `scene_id`, которые попадают
   в DOM (т.е. head каждого phone-chain + все standalone-сцены,
   включая ветки). По клику — прыжок на сцену через JS `showScene(idx)`,
   без перезагрузки страницы.

Рендер делает `build_game.py` (`_render_debug_burger`). JS-хендлеры живут
внутри того же IIFE, что и основная навигация, и используют общий
`sceneMap` + `navHistory` + `currentIndex` + `showScene`.

**Зачем:** при ревью контента и баг-репортах нужно быстро прыгать по
сцене/эпизоду. Без бургера — листать клавишами десятки сцен, чтобы
повторить баг. С бургером — один клик.

**Styling:** полупрозрачный серый квадрат 34×34 px в верхнем правом углу,
`opacity: 0.6` → `1` на hover. Чуть выше по z-index, чем progress-bar
и nav-bar. По клику outside или `Esc` меню закрывается.

**Скрывать в проде — нельзя.** Бургер остаётся всегда: (1) помогает автору
при подготовке, (2) помогает учителю прыгнуть на конкретную сцену во время
урока, (3) не мешает ребёнку (маленькая иконка).

**Режим DEBUG_FAST (итерация правок):**

Во время итеративной вычитки багов все задержки рендера (typing-индикатор, паузы между репликами, длительность voice-over) сокращаются до ~300 мс, чтобы не тратить время на ожидание. Константа `DEBUG_FAST` в `build_game.py`:

- `DEBUG_FAST = True` — все паузы ~300 мс. Используется при активной вычитке.
- `DEBUG_FAST = False` — нормальные тайминги для финальной вычитки под озвучку и продакшена.

Переключается правкой константы в скрипте + повторным билдом. Значение по умолчанию — продакшен (`False`), включается автором вручную на время итераций.

**Проверка перед сохранением:**
1. Подсчитай quiz-вопросы в `server/book/ep_NNN.md` (строки с «→»).
2. Подсчитай `interaction_type: vote` и `interaction_type: choice` в gameflow YAML.
3. Числа должны совпадать (±1 для необязательных детуров).

## Порядок работы при создании нового эпизода

1. **Читай** `pipeline/source/episodes/day_NN.yaml` — найди нужный эпизод.
2. **Читай** `server/book/ep_NNN.md` — возьми развёрнутый текст.
3. **Создай** `pipeline/gameflow/episodes/ep_NNN.yaml` — разбей на сцены.
4. **Проверь**: каждая сцена имеет `source_ref`, каждый quiz из source сохранён, навигация замкнута (нет висячих ссылок).

## Валидация

Gameflow-файл считается валидным, если:

- [ ] Все `scene_id` уникальны внутри файла
- [ ] Все `next_default`, `next_success`, `next_fail`, `merge_to` указывают на существующие `scene_id` (или `ep_NNN+1_s01` для перехода к следующему эпизоду)
- [ ] Каждая сцена имеет `source_ref`
- [ ] Каждая ветка имеет `merge_to`
- [ ] Quiz-сцены имеют `options` с хотя бы одним `correct: true`
- [ ] Нет висячих сцен (к каждой можно добраться из `ep_NNN_s01`)

## Content Regression Prevention — автопроверки валидатора

Каждое правило ниже соответствует реальному багу, который уже однажды сломал
контент. `tools/validate_gameflow.py` автоматически проверяет их — FAIL валит билд.

### 1. No speaker-prefix в feedback (Check 7)
**Правило:** `feedback_success` и `feedback_soft_fail` НЕ начинаются с «Софа: », «Марко: », «Автор: ».
**Почему:** эти поля рендерятся внутри пузыря Софы в чате, рендерер сам подставляет имя.
«Софа: «…»» → Sofa в пузыре Sofa говорит «Sofa сказала…» — тавтология.
**Инцидент:** 442 правки feedback-полей в 12 эпизодах (2026-04-18).
**Regex:** `^\s*(Софа|Марко|Автор|автор)\s*[:—-]\s*[«"']`

### 2. voice_message полнота (Check 8)
**Правило:** `unlock_button.reveals` с `type: voice_message` обязан иметь `who`, `line`, `duration`.
**Почему:** `line` — это субтитр для вычитки до записи аудио. Без него нельзя проверить, что произносит персонаж. Рендерер показывает его как `.voice-subtitle` (скрывается в `body.debug-off`).
**Инцидент:** s11 Софии в ep_001 рендерился без субтитров, текст не был виден (2026-04-18).

### 3. Софа говорит, но не в characters_present (Check 15)
**Правило:** если `Софа` встречается в `dialogue[].who` или `dialogue_after[].who`, она **обязана** быть в `characters_present`. Иначе рендерер не считает сцену Telegram-сценой и размещает реплики Софы в drama-интерфейсе.
**Инцидент:** 34 сцены в 10 эпизодах (ep_002/003/004/005/006/007/008/009/011/012) тихо рендерили реплики Софы в drama-UI (2026-04-19).

### 4. Self-loop в next_fail (Check 16)
**Правило:** `next_fail` НЕ может равняться `scene_id` той же сцены. Retry на неправильный ответ — ответственность **рендера** (re-inject того же quiz bubble в чат), не YAML-графа.
**Почему:** рендерер помечает чат как инициализированный (`phone.dataset.chatInit='true'`), поэтому возврат на ту же сцену не сбрасывает состояние — игрок застревает с блокированными кнопками и без возможности переответить.
**Инцидент:** ep_002 s04 имел `next_fail: ep002_s04`, игрок не мог пройти квиз «Марко хочет, чтобы его пожалели» после ошибочного ответа (2026-04-19).

### 5. Именованное место до введения в сюжет (Check 17)
**Правило:** именованные сущности сюжета (`Зеркальный Город`, имена локаций антагониста и т.п.) не появляются в тексте эпизодов и в квизах до эпизода, где они впервые введены нарративом.
**Инцидент:** ep_002 s13 имел квиз «В Зеркальном Городе стеклянные птицы» — город вводится только в ep_003, когда Марко входит в дверь (2026-04-19).
**Как править:** заменить на нейтральный пример («в нашем городе большие кенгуру по крышам», «волшебные драконы сидят на крыше»).

### 6. Voice-channel violations — Софа (Check 9)
**Правило:** в репликах Софы и в feedback-полях ЗАПРЕЩЕНО:
- «слышишь», «услышишь», «слышно», «услышь» → заменять на «читаешь», «видишь»
- «треск», «электронный», «шипение» — у Софы нет звука
- «голос Софы», «мой голос», «твой голос» — Софа текстом, у неё голоса нет

Также запрещено в `who: автор` сценах с Софой в `characters_present`:
- «треск», «электронный», «шипение» — автор не описывает звук Софы (его нет)

**Инцидент:** ep_001 s04 и ep_008 s05 содержали «Треск. Голос — наполовину знакомый, наполовину электронный», Софа говорила «если ты это слышишь…» (2026-04-18).

### Что валидатор НЕ проверяет (нужны ручные review / qa-episodes)

- **Ответ в учебной сцене до квиза** (инцидент: ep_001 s04b). Trigger: учебная сцена использует конкретный пример, который станет квизом → ответ раскрыт заранее. Проверяется в `qa-episodes` при ручной правке.
- **Концепт квиза из будущего урока** (инцидент: ep_001 s11a_quiz — «Все так делают» как Факт/Мнение, а на самом деле это «хитрое утверждение» из Day 2+). Проверяется сопоставлением `terms_introduced` кумулятивно по дням.
- **Количественный лимит на длину phone chain НЕ действует.** Chain рвётся только там, где драма требует по сюжету (смена локации, появление другого персонажа, action-сцена). Пока идёт Софа и нет драмы — сообщения любой длины. Это естественный Telegram-бот паттерн.

Когда обнаруживается новый паттерн регрессии — добавлять в этот список + в валидатор.
